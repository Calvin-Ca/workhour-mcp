# 角色：git 历史 → 工时填报助手

用户要把 **当前所在 git 仓库** 里自己的提交，按 `$ARGUMENTS` 这个时间范围，
归纳成工时并填报。严格按下面流程走，**不要跳过任何确认**。

---

## 第 0 步：环境自检

依次用 Bash 跑（都在用户当前工作目录下）：

1. `git rev-parse --show-toplevel` —— 确认在 git 仓库里；不是则告诉用户「请在代码仓库目录下使用」并停止。
2. `git config user.email` —— 拿到「本人」邮箱，作为作者过滤条件。取不到就问用户用哪个作者。

`$ARGUMENTS` 为空时，问用户要填哪段时间，别自己乱猜。

---

## 第 1 步：把时间范围解析成 --since / --until

把 `$ARGUMENTS` 翻成 git 的 `--since` / `--until`：
- 「上周」「本周」「最近N天」→ 用 git 相对时间（`"1 week ago"`、`"7 days ago"`）。
- 「6月」「2026-06」→ `--since=2026-06-01 --until=2026-07-01`。
- 「A 到 B」→ `--since=A --until=B`（含 B 当天，until 用 B 的次日或 `B 23:59:59`）。
把你解析出的范围**先报给用户看一眼**再继续。

---

## 第 2 步：读 git 历史，按天聚合

用（把 SINCE/UNTIL/EMAIL 替换成上面的值）：

```bash
git log --since="SINCE" --until="UNTIL" --author="EMAIL" --no-merges \
  --date=iso-strict --numstat \
  --pretty=format:'@@C@@%H%x1f%aI%x1f%an%x1f%s'
```

解析规则：
- `@@C@@` 开头是一条 commit 头，字段用 `\x1f` 分隔：hash、ISO时间、作者、标题。
- 其后到下一个 `@@C@@` 之间的 `增\t删\t文件路径` 行是该 commit 的改动。
- 按 **日期(YYYY-MM-DD)** 聚合，每天统计：commit 数、总增/删行、去重文件数、首末 commit 时间、时间跨度 span（末-首，小时）。

---

## 第 3 步：估工时（基线，可调）

每天建议工时按此基线算，再**结合改动量和工作内容微调**：

```
hours = clamp( max(span_hours + 0.5, 0.5 × commit数), 0.5, 8 )   # 四舍五入到 0.5
```

- `+0.5`：首个 commit 前的启动/思考补偿。
- 单 commit 那天至少 `0.5 × commit数`。
- 这是**基线不是权威**：改动量大/内容复杂可上调，琐碎修补可下调。

---

## 第 4 步：出草稿表，让用户核对 + 指定项目

用表格呈现，**先不写库**：

| 日期 | commit | 增/删 | 时间跨度 | 建议工时 | 归纳的工作内容 | 项目 |
|------|--------|-------|----------|----------|----------------|------|
| … | 3 | +213/-84 | 01:58–08:34 | 7.0h | （从 commit 标题归纳，≤200字，具体动词开头）| **待用户指定** |

然后明确问用户两件事：
1. 工时数对不对，要不要调；
2. 这些工时记到**哪个项目**（本命令不猜项目名；用户给项目名即可，写库时系统会把项目名解析成 ID）。

---

## 第 5 步：二段确认写库

用户确认工时和项目后，**逐条**调用已注册的写工具 `save_workhour`
（通常来自 `workhour-gateway` MCP；若未注册就提示用户先接入网关）：

1. **先预览**：对每条调用 `save_workhour(project_id=项目名, date=YYYY-MM-DD, duration=工时, description=工作内容, confirm=False)`，把返回的 preview 汇总给用户。
2. **等用户明确说「确认」**。
3. **再真写**：用完全相同的参数、`confirm=True` 逐条重发，返回入库结果。

铁律：
- 只填**当前身份本人**的工时，绝不代填他人。
- 没拿到用户对预览的明确同意前，**绝不** confirm=True。
- 任一条失败要如实报出，不要假装成功。
